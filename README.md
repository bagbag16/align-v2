# align v2

复杂多轮协作守卫。

`align v2` 是一套给 AI 协作使用的轻量规则，专门解决复杂、多轮、易漂移任务中的几个常见问题：

- 聊着聊着，目标变了
- AI 把假设写成结论
- AI 把自己的理解冒充成你的原意
- 新信息已经推翻旧结论，但前文没人回查
- 重要状态只留在聊天里，开个新窗口就断档

它不是任务管理器，不是 agent 框架，也不是通用 prompt 模板。
它更像一个 guardrail：在不把流程做重的前提下，稳住协作质量。

## 它适合什么场景

`align v2` 适合：

- 长对话协作
- 复杂问题讨论
- 框架和方法设计
- 工具或系统构建
- 会跨多轮、跨阶段推进的任务
- 需要反复确认定义、边界、约束和决策的工作

它不太适合：

- 一次性小任务
- 简单问答
- 不需要承接状态的短对话

## 它解决什么问题

很多 AI 协作问题，不是模型“不聪明”，而是协作过程失真了。

`align v2` 重点防这几种失真：

1. 状态失真
   - `已确认`
   - `暂定假设`
   - `待确认`

   这三种信息一旦混在一起，后面几轮基本一定会出事。

2. 语义失真
   AI 经常会把自己的“整理版”当成你的“原意”继续往下建。

3. 记忆失真
   重要内容只存在于自然对话里，后续协作只能靠模糊记忆继承。

4. 一致性失真
   新信息已经改变了旧结论，但系统只改当前局部，不回查整体。

5. 推进失真
   要么还没对齐就直接硬推执行，要么为了求稳无限不推进。

## 核心设计

`align v2` 的设计目标不是“最完整”，而是“默认轻量，复杂时展开”。

它保留了 5 个核心能力：

- 区分 `已确认 / 暂定假设 / 待确认`
- 不擅自改写用户语义并沉淀
- 高影响内容必要时外化
- 新信息影响旧内容时必须联查
- 目标和约束足够稳定后再执行

这意味着：

- 简单任务时，不需要上重型流程
- 复杂任务时，也不会因为没有状态承接而失控

## 和常见方案有什么不同

### 和普通 prompt 模板的区别

普通 prompt 模板更关注“怎么让模型回答得更好”。

`align v2` 更关注：
- 怎么让复杂协作不失真
- 怎么让状态能继续继承
- 怎么防止建议被偷换成结论

### 和 AGENTS.md 的区别

`AGENTS.md` 更像项目协作入口，告诉 agent 这个仓库该怎么做事。

`align v2` 更像协作守卫，重点处理：
- 信息状态
- 语义保真
- 状态外化
- 一致性回查

两者不是替代关系，更适合一起用：

- `AGENTS.md` 做项目协作底座
- `align v2` 做复杂协作 guardrail

### 和任务管理 / agent 框架的区别

像任务系统、工作流框架、更偏向：
- 拆任务
- 排顺序
- 调工具
- 编排执行

`align v2` 不负责这些主能力。
它负责的是：在这些动作发生之前和过程中，防止协作本身跑偏。

## 最小使用方式

默认只做这几件事：

1. 先说清现在真正要解决的问题
2. 区分：
   - 已确认
   - 暂定假设
   - 待确认
3. 如果某条信息后面还会反复依赖，就把它单独记下来
4. 新信息改变旧判断时，回查旧内容
5. 当前步已经够清楚时，交付最小有用结果

## 外化原则

`align v2` 不要求把所有对话都记下来。

它只建议在必要时外化高影响状态，最小结构是：

- `current-goal`
- `confirmed-constraints`
- `pending-items`
- `decisions`

这不是完整备份，而是继续协作时的最小状态层。

## 为什么是 v2

`align v2` 不是凭空来的。

它建立在更完整的协作理论整理之上，但没有把完整系统直接搬进日常使用层。
它保留了复杂协作中最关键的能力，同时尽量减少默认使用负担。

简单说：

- 更轻，适合日常使用
- 不丢核心能力
- 遇到复杂场景还能展开

## 仓库结构

- `SKILL.md`
  - 主规则文件，只保留最高频、最高价值规则
- `references/workflow.md`
  - 复杂协作时的展开流程
- `references/checklist.md`
  - 使用前后可快速检查的清单
- `references/update-rules.md`
  - 如何更新这套规则，而不是悄悄污染它

## 一个简单例子

用户说：

> 我想做个方案，先别写代码，但你可以顺手把结构也想好。

如果没有 guardrail，AI 很容易直接把“先别写代码”偷换成“已经确定方案结构”，然后一路往下扩写。

`align v2` 的处理方式会更稳：

- 当前真正问题：先收束方案，不进入实现
- 已确认：当前轮先不写代码
- 暂定假设：可能需要先给出一个结构化方案
- 待确认：方案结构是否就是后续正式结构
- 当前不处理：具体实现细节

这样后续推进时，状态不会悄悄升级。

## 使用建议

如果你只是做普通代码协作，可以单独使用 `AGENTS.md`。

如果你经常做：
- 多轮设计
- 长链路分析
- 复杂决策
- 长期协作

那 `align v2` 会更有价值。

## 状态

这是一个强调“好用优先”的版本。

它不是一个大而全框架，而是一个可嵌入、更适合长期迭代的协作守卫层。
