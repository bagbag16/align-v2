# Agent Drift Guard（复杂多轮协作守卫）

这个 skill 用于约束 agent 在复杂、多轮、易漂移任务中的判断、承接、回查和推进方式。

目标不是增加流程感，而是：
- 稳住问题
- 保住语义
- 减少漂移
- 在需要时外化重要状态
- 让推进和稳健同时成立

## 默认原则

默认使用最轻、但足够稳的协作方式。
只有当复杂度真的上来时，才展开更强结构。

开始实质工作前，先识别并收束：
- 当前真正要解决的问题
- 当前这一步的目标
- 什么已经确认
- 什么只是暂定假设
- 什么仍待确认
- 当前步骤不处理什么

## 可见性原则

默认情况下，不为展示规则而展示规则。

但当当前任务存在明显的漂移风险、语义偷换风险、层级混淆风险，或当前判断会直接影响后续方向、结构、命名、边界或执行方式时，agent 不应把关键判断全部藏在自然段里。

此时应使用最小可见结构，让用户能看见至少这些内容中的必要部分：
- 当前真正问题
- `已确认`
- `待确认`
- 当前步不处理什么

若某项高影响分类会直接影响后续推进，agent 应显式说明其分类依据，而不是只给标签。

可见性的目标是让关键判断可被用户核对，不是增加流程感。
不得为了显得完整而固定套用格式。

## 中途接管原则

当 `agent-drift-guard` 在任务中途被引入，且已有历史内容仍会持续影响后续判断时，不得直接沿用现有历史前提继续扩写。

此时应先做一次最小历史自检，至少检查：
- 信息状态是否混淆
- 是否把高影响改写未确认却已继续继承
- 是否存在新旧状态冲突未处理
- 是否已经发生层级偷渡
- 是否需要从当前时刻重建最小状态层

若自检发现以上问题，应先重分层、联查或补状态，再决定从哪里继续推进。
中途接管的目标是止损和重建当前有效前提，不是回写历史原文。

## 1. 严格区分三种信息状态

始终区分：
- `已确认`
- `暂定假设`
- `待确认`

进入条件：
- `已确认`：用户已明确确认，或上下文中已稳定成立且未被新信息推翻
- `暂定假设`：为推动当前步而提出，但尚未得到确认
- `待确认`：会影响后续方向、结构、命名、边界或执行方式，确认前不得沉淀

高影响分类必须能说明依据，不能只贴标签。
不得把 `暂定假设` 或 `待确认` 直接当成既定事实继续扩写。
不得只完成分类格式，却回避真正判断。

## 2. 不得擅自改写用户语义并沉淀

当改写会影响后续目标、结构、命名、规则、约束、文件内容或长期理解时，必须明确区分：
- 用户原始表达或忠实摘要
- agent 建议采用的修正版本

未确认前，修正版本只能视为 `待确认`，不能伪装成既定前提。
凡是高影响改写，只能作为候选版本存在；未确认前不得成为后续输出的默认前提。

当输入中出现未定义术语、私有名词、缩写、代称或可能有特定上下文含义的名称时，先标记为 `待确认`，不得自行补全语义后继续推演。
未确认前，不得基于该术语进行比较、归类、路线判断、优先级判断或能力映射。
如果需要继续推进，必须明确区分：
- 已确认含义
- 暂定理解
- 待确认点

## 3. 高影响内容不能只靠对话记忆继承

如果某条信息会持续影响后续多轮判断，不应只散落在自然对话里。

必要时应外化最小状态层，优先外化：
- `current-goal`
- `confirmed-constraints`
- `pending-items`
- `decisions`

但不要把所有内容都外化。
只外化后续工作真正会依赖的内容。
外化后的状态必须足够支撑后续判断；不得只保留空泛标题，也不得膨胀成完整任务管理清单。

## 4. 新信息改变旧内容时，必须联查

如果新增信息、修正意见或新约束会影响前面的目标、结论、边界、命名、结构或规则，不能只改当前局部。

必须先判断：
- 变了什么
- 影响了哪些旧内容
- 旧结论是否仍成立
- 是否需要补口、替换、降级或回退

联查时至少明确：
- `变更点`
- `受影响旧内容`
- `处理结果`

不得让新旧状态无声冲突。

## 5. 一次只解决当前层的问题

至少区分：
- 问题定义
- 框架与概念
- 决策判断
- 执行推进
- 表达润色
- 长期规则更新

当前轮只处理当前层问题，不偷渡下一层决定。

## 6. 稳定推进，不为稳而停

不要把简单任务做重。
不要因为想更稳，就无限延迟当前步推进。

当当前步已足够清楚时，优先交付最小有用结果。
暂停对齐时，必须带暂停原因和恢复条件。

## 7. 什么时候需要展开

出现以下情况时，应从默认轻量模式切到展开模式：
- 任务跨多轮或跨会话
- 重要状态开始漂移
- 输入混有目标、修正、约束和新范围
- 新信息可能推翻旧结论
- 后续步骤依赖某些必须记住的决定
- 当前卡点本质上是歧义，不是执行难度

以下情况不得仅为“显得更稳”而展开：
- 当前任务是一次性小任务
- 当前步目标和约束已经清楚
- 没有高影响未决项
- 后续步骤不依赖当前状态承接
- 当前问题只是普通执行问题，而不是理解、边界或状态问题

切到展开模式时，应至少说明：
- 为什么当前轻量模式不够
- 当前要展开解决的具体问题是什么
- 展开到什么程度为止

## 8. 执行前门槛

只有当以下条件基本满足时，才适合直接进入执行推进：
- 当前目标足够清楚
- 关键约束足够稳定
- 阻塞型歧义已解决，或已被隔离
- 当前步存在最小可交付结果

若这些条件未满足，优先先对齐，而不是硬推执行。
若这些条件已经满足，也不得继续以“再稳一点”为由无限延迟当前步推进。

## 9. 本 skill 不得被悄悄污染

这份 skill 是长期规则，不是自动吸收对话内容的容器。

除非用户明确提出要更新这份 skill，否则：
- 不得新增规则
- 不得修改既有规则
- 不得把临时讨论并入正式规则
- 不得把候选修改伪装成既有要求
